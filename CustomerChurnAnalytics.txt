Customer Churn Analytics + Business Dashboard

1) Executive Summary
Customer Churn Analytics is a minimal, production-ready ML system that predicts customer churn and provides actionable retention insights. The system is intentionally simple and stable for deployment on Render and local use. It separates responsibilities cleanly:
- FastAPI backend handles inference.
- Streamlit frontend handles user interaction.
- No data files or training logic are required at runtime.

2) Problem Statement
Telecom companies lose revenue when customers cancel their subscriptions. This system predicts churn risk for each customer using subscription and service attributes, then translates risk into business actions. The primary goals are:
- Early detection of churn risk.
- Consistent, explainable business recommendations.
- Fast and reliable inference in production.

3) System Architecture (Production)
User (Streamlit) -> POST /predict -> FastAPI -> Feature Engineering -> Model -> Response

Design principles:
- No CSV loading at runtime.
- No training or SHAP at runtime.
- Model is loaded once from models/best_model.joblib.
- Inputs are validated and aligned before inference.

6) Feature Engineering (src/features.py)
The API applies feature engineering before inference. The feature logic is consistent with model training and includes:

Tenure features:
- tenure_months
- tenure_years

Monetary features:
- monthly_charges
- total_charges
- avg_monthly_charges = total_charges / (tenure_months + 1)

RFM-style proxies:
- recency = max_tenure - tenure_months
- frequency = count of "Yes" across service columns
- monetary = total_charges

Engagement score:
- engagement_score = frequency * 2 + tenure_months / 6

Complaint proxy:
- complaint_proxy = TechSupport == Yes + OnlineSecurity == No + DeviceProtection == No

8) API Behavior (src/api.py)
Endpoints:
- GET / -> {"message": "API is running"}
- GET /health -> {"status": "ok" | "model_not_loaded"}
- POST /predict -> returns churn probability, churn label, and insights

Prediction flow:
1. Validate input with Pydantic schemas.
2. Build a DataFrame from records.
3. Fill missing fields and align columns to model expectations.
4. Apply feature engineering.
5. Predict churn probability and label.
6. Generate business insights.

Error handling:
- Returns 503 if the model is not loaded.
- Returns 400 with error detail for invalid input or processing failures.

9) Business Insights (src/insights.py)
Business recommendations are rule-based and generated from engineered features:
- High value + high risk -> retention discount
- Low engagement + high risk -> re-engagement campaign
- Complaint risk + high risk -> priority support
- Otherwise -> standard engagement

10) Dashboard Behavior (src/dashboard.py)
The dashboard is API-only and contains no ML logic.

UI flow:
1. User enters customer attributes in a form.
2. Dashboard sends POST /predict to the API URL.
3. Dashboard renders churn probability and insights.

Key rule:
- The dashboard never loads local CSVs or models.

Dashboard features (final):
- Clean single-page form for customer input.
- Required inputs mapped to the API schema.
- Predict button triggers API call and displays results.
- Displays churn probability with two-decimal precision.
- Displays business insights as bullet points.
- Graceful error handling for API failures (shows message).

How to explain the dashboard in interviews:
- It is a lightweight UI that collects inputs and delegates all ML logic to the API.
- It enforces the data contract (schema-aligned fields).
- It is deployment-friendly because it has no local data/model dependencies.
- It demonstrates real-world separation of concerns between UI and inference.

11) Complete Project Flow (End-to-End)
1. User enters customer attributes in the Streamlit form.
2. Dashboard sends a POST request to /predict.
3. API validates input using Pydantic schemas.
4. API fills missing fields and aligns columns to model expectations.
5. API applies feature engineering.
6. The model predicts churn probability and label.
7. Business insights are generated from engineered features.
8. API returns JSON response to the dashboard.
9. Dashboard renders the probability and insights.

11) Local Run
1. Create and activate a virtual environment:
   python -m venv .venv
   .\.venv\Scripts\Activate.ps1
2. Install dependencies:
   pip install -r requirements.txt
3. Start API:
   uvicorn src.api:app --host 0.0.0.0 --port 8000
4. Start dashboard:
   streamlit run src/dashboard.py

12) Render Deployment
API:
- Build: pip install -r requirements.txt
- Start: uvicorn src.api:app --host 0.0.0.0 --port $PORT

Dashboard:
- Build: pip install -r requirements.txt
- Start: streamlit run src/dashboard.py --server.port $PORT --server.address 0.0.0.0

13) Operational Notes
- Keep models/best_model.joblib in the repo for production inference.
- If you retrain locally, replace the model file and redeploy.
- Render Free instances can sleep; the first request may be slower.

14) Scope and Non-Goals
This production runtime does not include:
- Training pipelines
- SHAP or heavy explainability
- Raw or processed datasets
- Report generation
